# Prompt Cookbook — ascii-draw 用

ASCII Draw の AI 生成エンジンに「効く」プロンプトのパターン集。

## ⭐ 黄金ルール

1. **構成要素を最初に列挙**（誰が登場するか）
2. **矢印 `→` で関係を明示**（何が何にどう繋がるか）
3. **役割が違う要素はカテゴリ語を添える**（DB / API / UI / 外部 etc.）→ 自動色分けが当たる
4. **時系列・条件分岐は箇条書き**（番号 1. 2. 3. または順序語「次に」「並列で」）
5. **長すぎる説明は逆効果**（80〜180 文字が最も精度が高い）
6. **「縦長にしない」「横並び優先」と明記**するとレイアウトが安定

## 1️⃣ システムアーキテクチャ図

### Webアプリ標準3層
```
Webアプリのアーキテクチャ図: フロントエンド React → API Gateway → 3つのマイクロサービス (認証 / 注文 / 通知) → PostgreSQL DB。横並び優先。
```

### マイクロサービス + キャッシュ + キュー
```
マイクロサービス構成: クライアント → Nginx → API Gateway → 認証サービス・商品サービス・注文サービスの3つ並列 → 共通の Redis キャッシュ + RabbitMQ メッセージキュー → PostgreSQL。
```

### サーバーレス AWS
```
AWS サーバーレス構成: ユーザー → CloudFront → S3 静的サイト → API Gateway → Lambda 関数 → DynamoDB。並列で SQS → 別の Lambda → SES メール送信。
```

### モノレポ + デプロイ
```
モノレポのデプロイ構成: GitHub → GitHub Actions (lint, test, build の3並列) → Docker イメージ → ECR → ECS Fargate → ALB → ユーザー。
```

## 2️⃣ フローチャート / プロセスフロー

### ユーザー登録フロー
```
ユーザー登録フロー: 1.入力フォーム → 2.バリデーション (NG なら入力に戻る) → 3.DB 保存 → 4.確認メール送信 → 5.完了画面。菱形で分岐。
```

### 注文処理
```
EC 注文処理フロー: カートに追加 → ログイン (未ログインならログイン画面へ分岐) → 配送先入力 → 決済 (成功 / 失敗で分岐) → 注文確定 → 在庫減算 → 確認メール。
```

### 業務承認
```
経費精算の承認フロー: 申請者が起票 → 部長承認 (差戻しなら申請者へ) → 経理確認 (差戻しなら部長へ) → 振込 → 完了通知。差戻しの戻り線も描く。
```

## 3️⃣ シーケンス図 (時系列)

> シーケンス図は **本格的な ──→ 矢印縦並び**よりも、**横並びの呼出フロー**として描く方が ASCII では綺麗。

### OAuth 2.0 認可コードフロー
```
OAuth 2.0 認可コードフロー: ユーザー → クライアントアプリ → 認可サーバー (ログイン画面表示) → ユーザー認証 → 認可コード発行 → クライアントが認可コード+secretをトークンエンドポイントに → アクセストークン取得 → リソースサーバーへ API コール → ユーザーデータ取得。
```

### JWT 認証
```
JWT 認証シーケンス: 1.クライアントが /login に email/password POST → 2.サーバーが DB 照合 → 3.成功なら JWT 署名して返却 → 4.以降クライアントは Authorization: Bearer ヘッダで送信 → 5.サーバーは JWT 検証 → 6.OK なら API レスポンス。
```

---

## 4️⃣ データパイプライン / ETL

### 標準 ETL
```
データパイプライン: 1.S3 にファイル到着 → 2.EventBridge でトリガー → 3.Glue ETL (抽出 → 変換 → クレンジング) → 4.S3 DataLake (Parquet) → 5.Athena でクエリ → 6.QuickSight ダッシュボード。エラー時は SNS で通知。
```

### Lambda アーキテクチャ
```
Lambda Architecture: ソースデータ → Speed Layer (Kafka → Storm リアルタイム集計 → Cassandra) と Batch Layer (HDFS → MapReduce 日次集計 → HBase) の並列 → Serving Layer で統合 → 可視化。
```

### MLパイプライン
```
ML パイプライン: 1.データ収集 → 2.前処理 (欠損補完 / 正規化) → 3.特徴量エンジニアリング → 4.学習データ / 検証データ分割 → 5.モデル学習 (3モデル並列で評価) → 6.最良モデル選定 → 7.デプロイ → 8.推論 API → 9.モニタリング → 1へフィードバック。
```

## 5️⃣ ER 図 / データモデル

> ER 図は ASCII では限界がある。**関係性 (1対N / N対N) を矢印ラベルに書く**のがコツ。

### EC データモデル
```
EC サイトの ER 図: User (id, email, name) → 1対N → Order (id, user_id, total) → 1対N → OrderItem (order_id, product_id, qty) → N対1 → Product (id, name, price) → N対1 → Category (id, name)。中間テーブルも描く。
```

---

## 6️⃣ 状態遷移図 (state machine)

### 注文ステータス
```
注文の状態遷移図: 「未決済」が初期状態 → 決済成功で「処理中」→ 出荷で「発送済」→ 配達完了で「完了」。「未決済」からタイムアウトで「キャンセル」、「処理中」から手動キャンセルで「キャンセル」、「発送済」から不在票で「再配達待ち」→ 再配達で「発送済」に戻る。
```

### TCP 接続
```
TCP 接続の状態遷移: CLOSED → LISTEN (passive open) または SYN_SENT (active open) → SYN_RECEIVED → ESTABLISHED → FIN_WAIT_1 → FIN_WAIT_2 → TIME_WAIT → CLOSED。受動側は CLOSE_WAIT → LAST_ACK → CLOSED。
```

---

## 7️⃣ ネットワーク / インフラ

### VPC 構成
```
AWS VPC 構成: インターネット → IGW → Public Subnet (ALB) → Private Subnet (EC2 × 2 AZ) → DB Subnet (RDS Multi-AZ)。NAT Gateway 経由で Private Subnet から外部へ。
```

### Kubernetes クラスタ
```
Kubernetes クラスタ: Ingress → Service → Pod (Deployment) × 3 レプリカ → ConfigMap / Secret マウント → PV/PVC で永続ストレージ → 別 Namespace の Redis Pod へ。HPA で自動スケール。
```

## 部分修正のプロンプト例 (Selection-only mode)

範囲選択中は AI が **選択範囲だけを書き換える** ので、短い差分指示が効く：

| 状況 | 指示文 |
|-----|------|
| DB を別製品に変えたい | `DB を PostgreSQL に変えて` |
| サービスを 3 から 2 に減らす | `中段の 3 サービスを「Auth」「Order」の 2 つに統合` |
| 矢印の向き反転 | `Cache から Service への矢印を逆向きに` |
| ラベル追加 | `各矢印に「HTTPS」と注記` |
| エラー経路追加 | `右下に「失敗」分岐を追加して通知サービスへ` |

⚠️ 範囲選択せずに同じ指示を出すと **図全体** が書き換わる。必ずアプリ内で `V` キー → 直したい箱所をドラッグ → 入力 → `Ctrl+Enter` の手順で。

## ❌ うまくいかない指示パターン

| 悪い例 | なぜダメか | 直し方 |
|------|----------|------|
| 「いい感じの図を作って」 | 構成要素が指定されていない | 登場人物を最低3つ列挙する |
| 「綺麗で分かりやすくて...」 (装飾語多め) | LLM が要素を抽出できない | 装飾語を削って事実のみに |
| 「Aから始まって最終的にBになるシステム」(中間が曖昧) | 中間ノードを LLM が想像する | 中間も具体的に書く |
| 「縦長で1列にして」 | ASCII では縦長は読みづらい | 横並びを基本に |
| 「全部の機能を1枚に」(過剰盛り) | 70 ノード超えると破綻 | サブシステム単位に分割 |
